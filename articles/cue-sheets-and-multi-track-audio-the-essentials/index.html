Cue Sheets and Multi-Track Audio: The Essentials

By Logan Pierce • Published November 22, 2025

Cue sheets and multi-track audio often decide whether a disc-based game loads seamlessly or stutters at a track change. This guide aligns terminology, shows how audio tracks, indices, and gaps interact, and explains how a CUE file orchestrates them. By understanding the essentials around cue directives and multi-track layouts, you can choose between CUE plus BIN or compressed containers like CHD with confidence. The focus is practical: what preserves timing, what simplifies storage, and which quirks break compatibility on certain cores. You will leave with a simple framework to pick formats, verify layout integrity, and keep metadata predictable across tools.

Key takeaways

  • Cue sheets map tracks, gaps, and indices that control audio timing.
  • Use multiple tracks to preserve Red Book audio without merging silence.
  • Pick CUE plus BIN for editing flexibility and raw layout visibility.
  • Choose CHD to compress while keeping cues, gaps, and indices intact.
  • Validate with hashes and test tricky transitions to spot timing errors.

Concepts and definitions

Cue sheets describe the track map of a disc image, specifying file sources, track types, indices, and gaps so an emulator reproduces timing accurately. The core directives are FILE, TRACK, INDEX, and PREGAP or POSTGAP, which together define where data and audio begin. For mixed-mode games, TRACK 01 MODE1/2352 commonly holds data, while subsequent TRACK xx AUDIO entries hold Red Book audio. Treat the CUE as the authoritative table of contents. If it points to the wrong file or offsets, you can get missing music, clicks at transitions, or hangs. This matters because many cores trust the cue layout over internal guesses. Good practice is to keep filenames consistent and ensure the CUE references exact case and extensions.

Multi-track audio exists to preserve discrete Red Book tracks with real boundaries rather than baking everything into a monolithic stream. Each track can carry its own index 00 lead-in and index 01 start, which some games use to trigger events or loop music precisely. A typical audio track begins at index 01, but index 00 can represent pre-gap silence or hidden lead-ins. Emulators that honor indices will navigate those markers for seamless playback. If indices are flattened or gaps are misrepresented, loops can restart off-beat or cinematics desync slightly. While many titles are tolerant, rhythm-heavy or FMV-laden releases expose timing drift quickly, so faithful track segmentation prevents subtle regressions.

Formats interplay with cue sheets differently and that shapes your workflow. BIN pairs with CUE by storing raw sectors while the CUE maps structure, giving transparent control over sectors, pregaps, and mixed modes. ISO alone lacks audio track detail and is poor for mixed-mode discs, so it often needs supplemental cues or conversion. CHD compresses the set while keeping layout metadata, improving storage and integrity checks through embedded hashes. Each choice trades editability against convenience. If you frequently repair indices or adjust gaps, CUE plus BIN keeps changes simple. If you prioritize compact archives with consistent verification, CHD offers strong compression while retaining the cue semantics.

Frameworks and decision lenses

A reliable decision lens starts with intent: archival fidelity, library simplicity, or rapid editing. For archival fidelity, prioritize formats that maintain sector sizes, indices, and pregaps exactly, since these details affect music loops, data loads, and seek behavior. For simplicity, favor containers that bundle data with cue semantics so renames and relocations do not break references. For editing, use layouts where the CUE remains human-readable and track assets live as discrete files. Map these goals to outcomes. CHD aligns with archival and simplicity by embedding layout and hashes, while CUE plus BIN aligns with editing since you can reindex or split tracks without re-encoding. Reassess when moving between frontends that differ in CUE parsing quirks.

Timing integrity depends on how gaps and indices are represented, so use a lightweight test suite before bulk conversions. Build a short scenario list that includes back-to-back audio transitions, an FMV start, and a known loop point, then run it on your target core. If you hear a pop at a transition, suspect missing index 00 or an incorrect pregap duration. If FMV launches late, verify track order and data track mode. For libraries with varied provenance, compare behavior before and after converting to CHD to ensure parity. When differences appear, the safest fix is to correct the cue mapping rather than normalize audio, since destructive edits can mask the root cause.

Organizational choices reduce long-term friction as collections grow. Keep cue sheets co-located with their referenced files and avoid path-dependent entries that break when folders move. Prefer relative paths in FILE lines and stable naming that mirrors the disc label. Save verification artifacts next to images so audits are repeatable across machines. Looking ahead

The next phase is to standardize a small validation routine before large conversions, because catching layout errors early prevents library-wide regressions. Start by choosing two titles with loop-heavy soundtracks and one with a cinematic intro to probe index and pregap handling. Measure success by the absence of clicks at joins and by consistent start times across cores you actually use. When parity holds, apply your chosen format across similar discs. If not, pause and correct the cue mapping, then re-test. This rhythm builds trust in your workflow without overfitting to a single emulator’s quirks, and it creates a repeatable gate for future imports.

Library hygiene benefits from shared references and well-documented audits, so capture the reasoning behind each conversion in a small text note. Record source hashes, the tool and version used, and any cue adjustments. These context notes make it easier to reproduce results when you migrate storage or rebuild indexes. Understand what BIOS files do in emulators, why they are needed, and how they impact compatibility to separate platform startup problems from disc layout issues. When audio anomalies surface, confirm hashes first, then examine the cue for file case mismatches or misplaced gaps. If a conversion to CHD changes behavior, compare with the original CUE plus BIN to isolate the difference. This deliberate pace preserves audio quality while steadily reducing friction, so your future sessions focus on playing rather than troubleshooting.

Key terms

  • Binaural Audio — A playback method that uses HRTF to simulate 3D sound over headphones.
  • BIOS — Firmware code required to boot some systems in emulators.
  • CHD — A compressed format for disc or HDD images used by emulators.
  • Emulator Core — The component that emulates a specific system.
  • Firmware — Permanent software that controls a device's hardware.
  • ISO Image — A sector-by-sector copy of an optical disc.
  • MD5 — A 128-bit hash function used for file verification.
  • Pacing — The rise and fall of intensity over time.
